Complete Beginner's Guide to Git & GitHub

15 min read Beginner Updated July 2026

What is Version Control?

Version control tracks changes to files over time. It lets you:

  • Save different versions of your work
  • Undo mistakes
  • Collaborate with others
  • Work on features without affecting the main code
Think of it like a save point in a video game

Just like you save your game before a boss fight, you save snapshots of your code before making risky changes. If something breaks, you can always go back to your last save point.

What is Git?

Git is a version control system that runs on your computer. It tracks changes in your files and lets you create snapshots (called commits) at any point.

Git is local

Git runs entirely on your computer. You don't need an internet connection to use it. GitHub is where you store your Git projects online.

What is GitHub?

GitHub is a website where you store your Git projects online. It adds collaboration features like pull requests, issues, and team management.

The Git Workflow (Simple Version)

Git Workflow 
1. Edit files on your computer
2. Tell Git which files to track (stage)
3. Save a snapshot (commit)
4. Upload to GitHub (push)

Step 1: Install Git

  1. Download from git-scm.com
  2. Run the installer with default settings
  3. Open Git Bash from the Start menu
Bash
brew install git

Or download from git-scm.com

Bash
sudo apt-get install git   # Ubuntu/Debian
sudo dnf install git        # Fedora
Verify your installation

Open a terminal and run git --version. You should see something like git version 2.40.0.

Step 2: First-Time Setup

Tell Git who you are. This information is attached to every commit you make.

Bash
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"
git config --list
Use your GitHub email

Use the same email you registered with on GitHub so your commits are linked to your account.

Step 3: Create Your First Repository

Bash
mkdir my-first-repo
cd my-first-repo
git init

You should see:

Output
Initialized empty Git repository in /path/to/my-first-repo/.git/

The .git folder is where Git stores all its tracking data. Think of it as Git's database for your project.

Steps 4-6: Create, Stage, and Commit a File

Bash
echo "# My First Repository" > README.md
git add README.md
git commit -m "My first commit"
Output
[main (root-commit) a1b2c3d] My first commit
 1 file changed, 1 insertion(+)
 create mode 100644 README.md
Staging explained

Think of staging as putting items in a box before shipping. git add puts files in the box, and git commit ships it. You can add multiple files before committing.

Steps 7-8: Check Status & View History

Bash
git status
git log
  • git status — Shows what files have changed and what's staged
  • git log — Shows all commits with their messages, authors, and dates

Steps 9-10: Connect to GitHub & Push

Create a repository on GitHub, then run:

Bash
git remote add origin https://github.com/yourusername/my-first-repo.git
git push -u origin main
Your code is now on GitHub!

Refresh your GitHub repository page — you should see your README.md file with the commit message.

Daily Workflow

Git Workflow
1. Pull latest changes
   git pull

2. Make changes to files

3. See what changed
   git status

4. Stage changes
   git add .

5. Commit changes
   git commit -m "Describe your changes"

6. Push to GitHub
   git push

Common Mistakes (And How to Fix Them)

Committed to wrong branch
git reset HEAD~1
git checkout correct-branch
git cherry-pick HEAD@{1}
Wrong commit message
git commit --amend -m "New message"
Added the wrong file
git restore --staged filename

Best Practices

  • Commit often — Small, frequent commits are easier to understand and revert
  • Write good commit messages — Start with a capital letter, use imperative mood (e.g., "Add login feature" not "Added login feature")
  • Pull before you push — Always git pull first to avoid conflicts
  • Don't commit sensitive data — Never commit passwords, API keys, or personal information
  • Use .gitignore — Tell Git which files to ignore (build files, dependencies, OS files)
