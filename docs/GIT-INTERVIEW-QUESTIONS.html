Read each question, try to answer it yourself, then expand to see the solution. Practice explaining concepts out loud — interviews test communication as much as knowledge.

Beginner Questions

Git is a distributed version control system that tracks changes in source code during software development. It allows multiple developers to work on the same project simultaneously, maintains a complete history of changes, and enables branching and merging. It's used because it's fast, distributed (every developer has a full copy), handles large projects well, and has become the industry standard.

Git is the version control system that runs locally on your computer. GitHub is a cloud platform that hosts Git repositories online, adding collaboration features like pull requests, issues, actions, and project management. You can use Git without GitHub, but GitHub extends Git's capabilities for team collaboration.

A commit is a snapshot of your repository at a specific point in time. Each commit has a unique hash (e.g., a1b2c3d), an author, a timestamp, and a message describing the change. Commits form the building blocks of your project history and allow you to track exactly what changed and when.

A branch is a parallel version of your repository. It allows you to work on features, bug fixes, or experiments independently without affecting the main codebase. Once the work is complete and tested, you merge it back into the main branch. This is the foundation of Git's collaboration model.

git fetch downloads changes from the remote repository but does not merge them into your local branch. It lets you review changes before incorporating them. git pull is a shortcut that does git fetch followed by git merge, immediately integrating the remote changes into your local branch.

Intermediate Questions

When Git cannot automatically merge changes, it marks the conflict in the affected file:

<<<<<<< HEAD
your current changes
=======
incoming changes
>>>>>>> feature-branch

To resolve: 1) Edit the file to keep the desired code 2) Remove the conflict markers 3) Run git add <file> 4) Run git commit to complete the merge.

Rebasing rewrites commit history by applying your branch's commits on top of another branch. Use it to maintain a linear history and avoid merge commits. Rule of thumb: rebase before sharing (local cleanup), merge after sharing (public history). Never rebase shared branches.

git reset moves the HEAD pointer backward, optionally discarding commits (use with caution on shared branches). git revert creates a new commit that undoes a previous commit — safe for shared history. git restore is for discarding unstaged changes or unstaging files.

The staging area (also called the index) is an intermediate space between your working directory and the repository. It lets you selectively choose which changes to include in your next commit. This allows you to split unrelated changes into separate commits for cleaner history.

A Pull Request (PR) is a GitHub feature that lets you propose changes from one branch to another. Team members can review the code, leave comments, request changes, and approve the PR. Once approved, the PR is merged. PRs are the standard way to collaborate on GitHub.

Advanced Questions

HEAD is a pointer to the current commit/branch you're on. The working tree is your actual files on disk (what you edit). The index (staging area) is the proposed next commit. The flow is: Working Tree → git add → Index → git commit → Repository (HEAD).

Git stores data as objects in the .git/objects directory. There are four types: blob (file content), tree (directory listing), commit (snapshot with metadata), and tag (named reference). Each object is identified by a SHA-1 hash of its content. Files are stored as blobs, directories as trees, and commits point to trees with parent references.

The reflog (reference log) records every movement of HEAD. If you accidentally delete a branch or reset too far, you can recover with: git reflog to find the lost commit hash, then git checkout <hash> to recover it, or git branch recovered <hash> to create a new branch pointing to it.

Fast-forward merge happens when the target branch hasn't diverged — Git simply moves the pointer forward. Three-way merge creates a new merge commit when branches have diverged, combining changes from both branches and their common ancestor. Using --no-ff forces a merge commit even when fast-forward is possible.

Use interactive rebase: git rebase -i HEAD~n (where n is the number of commits to look back). In the editor, change pick to squash for commits you want to combine. This is useful for cleaning up messy commit history before merging a feature branch.

GitHub-Specific Questions

GitHub Actions automates workflows directly in your repository. You define workflows in .github/workflows/*.yml files. Workflows are triggered by events (push, PR, schedule) and run on GitHub-hosted or self-hosted runners. Common uses: running tests, deploying code, publishing packages, and automating releases.

A fork creates a copy of an entire repository under your GitHub account, allowing you to freely experiment without affecting the original. A branch is a parallel version within the same repository. Forks are used for external contributions (open source), while branches are used for internal team collaboration.

Scenario-Based Questions

1) git log to note the commit hash 2) git checkout correct-branch 3) git cherry-pick <hash> 4) git checkout wrong-branch 5) git reset HEAD~1 to remove the commit. This moves the commit to the correct branch and rewinds the wrong one.

Use git stash to save your work-in-progress temporarily. Then switch to the urgent branch, do the work, commit it, switch back, and use git stash pop to restore your previous changes. This is the standard workflow for handling interruptions.
