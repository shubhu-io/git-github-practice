Learning Path for Git & GitHub

20 min read Beginner

A complete 4-week structured guide to mastering Git and GitHub — from your first commit to advanced automation.

Prerequisites

No prior Git knowledge needed. You just need a computer with internet access and basic terminal/command-line familiarity.

Week 1: Git Basics

Day 1-2: Installation & Setup

Learning Objectives:

  • Understand what version control is and why Git dominates the industry
  • Install Git on your operating system
  • Configure your Git identity and preferences

What is Git? Git is a distributed version control system that tracks changes in files. It allows multiple developers to work on the same project simultaneously without overwriting each other's work.

OSInstructions
WindowsDownload from git-scm.com or run winget install Git.Git
macOSRun brew install git or download from git-scm.com
LinuxRun sudo apt install git (Ubuntu/Debian) or sudo dnf install git (Fedora)
Bash
git config --global user.name "Your Name"
git config --global user.email "your@email.com"
git config --global init.defaultBranch main
git config --global color.ui auto

Day 3-4: Basic Commands

Learn the core Git commands you'll use daily. Start with the Beginner's Guide for a step-by-step walkthrough.

Bash
mkdir my-project && cd my-project
git init
echo "# Hello Git" > README.md
git add README.md
git commit -m "Initial commit"
Output
Initialized empty Git repository in my-project/.git/
[main (root-commit) abc1234] Initial commit
 1 file changed, 1 insertion(+)
Pro Tip

Use git status frequently. It's the most useful command for understanding what's happening in your repository.

Day 5-7: First Repository & Push to GitHub

Create a GitHub account, create your first remote repository, and push your local code.

Bash
git remote add origin https://github.com/yourusername/my-project.git
git branch -M main
git push -u origin main

Week 2: Branching & Merging

Day 1-2: Creating & Switching Branches

Branches let you work on features without affecting the main codebase. Think of them as parallel universes for your code.

Branches allow parallel development. Each feature gets its own line of development.
Bash
git branch feature-login          # Create branch
git checkout feature-login        # Switch to it
git checkout -b feature-login     # Create & switch (shorthand)

Day 3-4: Merging Branches

When a feature is complete, merge it back into the main branch.

Bash
git checkout main
git merge feature-login

Day 5-7: Merge Conflicts & Resolution

Conflicts happen when two branches modify the same file. Git marks the conflict and asks you to resolve it.

File With Conflict
<<<<<<< HEAD
console.log("Hello from main");
=======
console.log("Hello from feature");
>>>>>>> feature

Edit the file to keep the desired version, then git add and git commit.

Common Mistake

Don't use git push --force on shared branches. It can erase other people's work. Use git push --force-with-lease instead — it's safer.

Week 3: Collaboration

Day 1-2: Remote Repositories

Learn how to sync your local repository with GitHub using push, pull, and fetch.

Bash
git clone https://github.com/user/repo.git  # Download a repo
git pull                                      # Get latest changes
git push                                      # Upload your changes
git fetch                                     # Check for changes without merging

Day 3-4: Pull Requests

Pull requests (PRs) are the heart of GitHub collaboration. They let you propose changes, discuss them, and merge them.

  1. Push a feature branch to GitHub
  2. Open a Pull Request on GitHub.com
  3. Request reviews from team members
  4. Address feedback with additional commits
  5. Merge the PR when approved

Learn more in the Pull Request Guide.

Day 5-7: Fork Workflow

Forking creates your own copy of someone else's repository. It's the standard workflow for open source contributions.

Week 4: Advanced Topics

Day 1-2: Stashing & Rebase

Stashing temporarily saves uncommitted changes so you can switch branches. Rebasing rewrites commit history for a cleaner project timeline.

Bash
git stash                 # Save changes temporarily
git stash pop             # Restore saved changes
git rebase main           # Reapply commits on top of main
Never rebase shared branches

Rebasing rewrites history. Only rebase branches that you haven't shared with others. Use merge for shared branches.

Day 3-4: Cherry-pick & Tags

Cherry-pick applies a specific commit from one branch to another. Tags mark specific points in history (usually releases).

Bash
git cherry-pick abc1234    # Apply a specific commit
git tag v1.0.0             # Tag the current commit
git push --tags            # Push tags to GitHub

Day 5-7: GitHub Actions & Automation

GitHub Actions lets you automate workflows like running tests, deploying code, and releasing packages.

YAML
name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm install && npm test
